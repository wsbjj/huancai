# 首期核心模块实现规格

目标：以当前 `index-1772524344345-ChF8viLm.js` 的交互逻辑为参照，当前文档已补齐这 15 个模块的实现规格：

- `c3` 批量生图
- `c15` 整套穿搭
- `c11` 买家秀制作
- `c4` 平铺转3D
- `c5` 裂变套图
- `c7` 换背景
- `c8` 生成细节
- `c12` 穿搭转平铺
- `c14` 模特图转白底
- `c19` AI改款
- `c20` 爆款裂变
- `c21` 服装换料
- `c22` 高清放大
- `c23` 复刻姿势
- `c9` 历史记录

其中：

- 前 10 个是首期服装工作台核心生成模块
- `c20` / `c21` / `c22` / `c23` 是第二阶段能力增强模块
- `c9` 是跨模块复用的历史工作台模块

这 15 个模块的规格已经足够定义完整的服装工作台范围，也能覆盖：

- 批量生成
- 单图编辑 / 改款
- 背景替换
- 商品局部细节生成
- 模特图商品提取白底
- 穿搭图转平铺
- 爆款参数裂变
- 面料替换
- 图像高清增强
- 姿势复刻
- 单图换装
- 多图组合换装
- 模板 / 模特库选择
- 临时模特上传
- prompt 编排
- 结果列表轮询
- 历史记录筛选 / 批量下载 / 删除

## 2. 共用技术架构

这 14 个生成 / 编辑模块都建议复用统一架构；`c9` 历史记录则复用同一套 `generated-results` 数据层：

### 前端

- `React + Vite + TypeScript`
- 左侧参数区
- 中间工作区 / 模板选择区
- 右侧结果列表区

### 后端

- 统一任务入口：`POST /api/generate-image`
- 统一处理中记录：`POST /api/generated-results`
- 统一历史查询：`GET /api/generated-results`
- 历史记录删除：`DELETE /api/generated-results/:id`
- 统一图片上传：`POST /api/generated-results/upload-image`

### 统一任务载荷

以下载荷适用于 14 个生成 / 编辑模块：

```ts
type GenerateImagePayload = {
  resultId: string;
  imageUrl: string;
  instruction: string;
  garmentImageBase64?: string;
  fabricImageBase64?: string;
  innerWearImageBase64?: string;
  backgroundImageBase64?: string;
  aspectRatio?: string;
  imageSize?: string;
  originalWidth?: number;
  originalHeight?: number;
  useRawPrompt?: boolean;
  skipCompress?: boolean;
};
```

## 3. 模块规格

### `c3` 批量生图

#### 产品目标

批量把多张服装图套到多张模特图上，生成电商模特展示图。

#### UI 结构

左侧：

- 上传服装图，支持批量
- 上传临时模特，支持批量
- 上传面料图，可选
- 尺寸比例
- 选择变体数量
- 可选自定义提示词

右侧：

- 结果列表
- 多选下载

#### 输入项

必填：

- 服装图数组
- 模特来源
  - 模特库照片，或
  - 临时模特图片数组

可选：

- 面料图
- 自定义 prompt
- 尺寸比例
- 变体数量

#### prompt 规则

默认 prompt：

`帮我把这个衣服换到这个模特身上，保证模特的脸型、细节、姿势、等不做任何改变。`

若开启“面料特写”，则追加：

`面料特写，让呈现出来的换装更清晰。`

每个结果都会再追加：

`- 变体n/b`

#### 任务展开规则

任务数：

- `模特数 x 服装数 x 变体数`

每个任务：

- `imageUrl = 模特图`
- `garmentImageBase64 = 服装图`
- `fabricImageBase64 = 面料图（可选）`
- `instruction = prompt + 变体后缀`

#### 历史数据特征

- `originalPhotoId` 通常是 `temp-model-*`
- `modelId = temp-model`
- `styleId = temp-style`

#### 平台实现建议

这个模块应优先做，因为它最能验证：

- 批量任务拆分
- 点数按任务数扣减
- 并行任务提交
- 结果轮询

### `c15` 整套穿搭

#### 产品目标

把上装、下装、内搭、鞋包等多件服装同时穿到一个模特身上。

#### UI 结构

左侧：

- 上传模特图
- 上传上装，可选
- 上传下装，可选
- 上传内搭，可选
- 上传鞋子或包包饰品，可选，支持多张
- 尺寸比例

右侧：

- 生成结果列表

#### 输入项

必填：

- 模特图

至少一项：

- 上装
- 下装
- 内搭
- 鞋子 / 包包饰品

#### prompt 规则

prompt 按已上传部件动态拼装：

- 基础句：
  - `帮我把这个衣服换到这个模特身上，保证模特的脸型、细节、姿势、等不做任何改变。`
- 若上传了部件：
  - `需要换上：上装、下装、内搭（里面的衣服）...`
- 若上传内搭：
  - 追加“内搭必须保持原样，必须使用我上传的内搭图片”
- 若上传鞋包：
  - 追加鞋包搭配说明
- 尾部固定：
  - `保证第一张图片的模特脸部/表情/姿势/眼神所有细节不做变动，4k,专业拍摄，大片。`

#### 任务载荷映射

- `imageUrl = 模特图`
- `garmentImageBase64 = 上装`
- `fabricImageBase64 = 下装`
- `innerWearImageBase64 = 内搭`
- `backgroundImageBase64 = 第一张鞋子 / 包包饰品`

#### 现有行为限制

虽然 UI 提示支持多张鞋包，但当前前端真正发给后端的只有第一张鞋包图。

#### 平台实现建议

第一期可以保持和现站一致：

- 先只发第一张鞋包图

第二期再升级为：

- 后端支持多附件数组

### `c11` 买家秀制作

#### 产品目标

把白底服装图生成成适合电商平台和内容平台展示的买家秀图。

#### UI 结构

左侧：

- 上传服装图
- 上传临时模特，支持批量
- 选择参考模特库
- 尺寸比例

中间：

- 参考模特列表 / 参考照片区域
- 可上传“参考模特”进入专属库

右侧：

- 结果列表

#### 输入项

必填：

- 服装图

二选一：

- 参考照片
- 临时模特

#### prompt 规则

固定默认 prompt：

`帮我把这件衣服穿到这个模特身上，不要改变衣服的细节和面料，不要改变模特的脸，我是作为买家秀在天猫和发小红书笔记，请给我一张好看的买家秀照片。`

#### 任务规则

每个选中的参考对象生成一个任务：

- `imageUrl = 模特图 / 参考照片`
- `garmentImageBase64 = 服装图`
- `instruction = 固定买家秀 prompt`

#### 特殊能力

当前前端允许把“参考模特”存入一个专门分类：

- `categoryId = buyer-show-reference`

这说明平台可以有“业务专用模特库”概念。

#### 平台实现建议

这个模块非常适合做成可运营能力：

- 支持沉淀参考模特库
- 支持后台审核 / 分类

### `c4` 平铺转3D

#### 产品目标

把服装平铺图或挂拍图转成真人上身效果图。

#### UI 结构

左侧：

- 上传衣服图片
- 上传挂拍图，可选
- 提示词，可选
- 选择模特库，非必填
- 尺寸比例

中间：

- 模特 / 模板卡片选择区

右侧：

- 生成结果列表

#### 输入项

必填：

- 平铺服装图

二选一或组合：

- 自定义挂拍 / 参考图
- 固定模特模板 / 模特卡片

#### prompt 规则

默认 prompt 不是简单换装，而是偏“挂拍形态复刻”：

- 典型历史 prompt：
  - `完全复刻参考图2 的挂拍形态...`
- 也存在更简单版本：
  - `生成白底3d图`

#### 任务映射

- `imageUrl = 平铺图`
- `garmentImageBase64 = 挂拍图 / 模板参考图`
- `instruction = 平铺转3D prompt`

#### 平台实现建议

第一期不追求“模板运营后台”：

- 先做固定模板 + 上传参考图

第二期再加：

- 模特模板管理
- 分类模板库

### `c5` 裂变套图

#### 产品目标

基于一张效果图，批量生成不同角度、动作、场景的套图。

#### UI 结构

左侧：

- 上传效果图
- 上传临时模特，可选，支持批量
- 选择模特库 / 风格 / 照片
- 尺寸比例
- 自定义提示词，可选

右侧：

- 结果列表

#### 输入项

必填：

- 效果图

二选一：

- 临时模特数组
- 已有模特库照片

#### prompt 规则

默认 prompt：

`图片A模仿图片B的姿势动作（要求动作姿态和状态一模一样）保证衣服的细节和人物的姿势以及脸，皮肤不会变化。`

如果用户输入了自定义 prompt，会在后面自动追加保护条件：

- `保证衣服的细节和人物的姿势以及脸，皮肤不会变化。`

#### 任务映射

- `imageUrl = 模特图 / 参考照片`
- `garmentImageBase64 = 效果图`
- `instruction = 裂变套图 prompt`

#### 平台实现建议

这个模块和换背景复用了当前前端里同一套“批量参考图任务”逻辑。

所以新平台建议抽公共模块：

- `reference-photo-based-generator`

### `c7` 换背景

#### 产品目标

把主体图或模特图的背景替换成目标背景，主体本身不变化，且融合自然。

#### UI 结构

左侧：

- 上传主体图 / 模特图
- 上传背景图
- 背景相似度滑杆
- 自定义提示词，可选
- 尺寸比例

中间：

- 若接入模特库，可复用参考照片选择区

右侧：

- 结果列表

#### 输入项

必填：

- 主体图 / 模特图
- 背景图

可选：

- 自定义 prompt
- 背景相似度
- 尺寸比例
- 临时模特或模特库照片

#### prompt 规则

默认 prompt 是固定模板，并插入相似度要求：

- 典型历史 prompt：
  - `将第一张图片的背景替换为第二张图片的背景，保持第一张图片的主体（人物、物品等）不变，背景自然融合。输出尺寸以上传的模特图的尺寸输出，不能改变模特的大小。`
- 低到中等相似度时追加：
  - `背景相似度要求：50%，适度融合背景。`
- 高相似度时追加：
  - `背景相似度要求：100%，尽可能保持背景的细节和特征。`

如果用户填写自定义 prompt，建议作为前缀，再自动补保护条件和相似度后缀。

#### 任务映射

- `imageUrl = 主体图 / 模特图`
- `backgroundImageBase64 = 背景图`
- `instruction = 换背景 prompt`

#### 历史数据特征

- `originalPhotoId` 常见为 `c7-model-*`
- `modelId = temp-model`
- `styleId = temp-style`

#### 平台实现建议

这个模块和 `c5` 很适合抽成同一个“参考图驱动型”任务框架：

- 前端只切换 prompt 构造器
- 后端复用同一条生成链路
- 相似度滑杆只作为 prompt 参数，不必单独设计底层协议

### `c8` 生成细节

#### 产品目标

基于一张商品主图，批量生成口袋、走线、领口、袖口等商品细节图。

#### UI 结构

左侧：

- 上传商品白底图
- 多选内置细节模板
- 添加 / 管理自定义细节模板
- 尺寸比例

中间：

- 模板列表区
- 自定义模板维护区

右侧：

- 按细节模板生成的结果列表

#### 输入项

必填：

- 商品主图
- 至少一个细节模板

可选：

- 自定义模板
- 尺寸比例

#### prompt 规则

该模块的 prompt 不是用户自由输入，而是以模板为主：

- 每个细节模板对应一条独立 prompt
- 内置模板示例：
  - `展示长裤缝线的细节，包括缝线工艺、走线方式、线迹细节，保持白底背景，专业拍摄，高清细节。`
  - `展示长裤口袋的细节，包括口袋形状、开口方式、缝线处理，保持白底背景，专业拍摄，高清细节。`
  - `展示衬衫领口的细节，包括领型、缝线、结构细节，保持白底背景，专业拍摄，高清细节。`

用户新增的自定义模板，也应直接落成一条模板 prompt。

#### 任务展开规则

任务数：

- `选中的细节模板数 x 主图数`

每个任务：

- `imageUrl = 商品主图`
- `instruction = 模板 prompt`

#### 现有接口特征

当前前端围绕以下资源做模板沉淀与图片存储：

- `detail-references`
- `custom-detail-templates`
- `detail-generation-image`

#### 平台实现建议

这个模块本质是“模板扇出生成器”，建议平台单独沉淀：

- `detail_templates` 表
- 系统模板 / 用户模板双层结构
- 一次提交自动展开成多条生成任务

### `c12` 穿搭转平铺

#### 产品目标

把真人穿搭图自动拆解为平铺商品展示图，并统一输出纯白背景。

#### UI 结构

左侧：

- 上传一张或多张穿搭图
- 尺寸比例

右侧：

- 平铺结果列表

#### 输入项

必填：

- 至少一张穿搭图

可选：

- 尺寸比例

#### prompt 规则

该模块使用固定长 prompt，不依赖用户自由编排，核心目标是：

- 自动识别图中的服装与配饰
- 拆解主要单品
- 以纯白背景平铺方式输出
- 尽量统一为 `800x800`

平台实现时建议把这段 prompt 固化为版本化模板，不在前端直接散写。

#### 任务映射

- `imageUrl = 穿搭图`
- `instruction = 固定穿搭转平铺 prompt`
- `originalWidth / originalHeight` 第一版可默认收敛到 `800`

#### 历史数据特征

- `originalPhotoId` 常见为 `uploaded-garment-*`
- `modelId` / `styleId` 通常为空

#### 平台实现建议

第一期可以把它当作简单“单图上传 -> 单图生成”模块：

- 不先做复杂编辑交互
- 先固定输出白底平铺图
- 后续再扩到多单品拆分下载

### `c14` 模特图转白底

#### 产品目标

从模特上身图中提取商品主体，生成适合电商使用的纯白背景产品图。

#### UI 结构

左侧：

- 上传模特图
- 选择提取部位：上衣 / 下装 / 鞋子

右侧：

- 按部位输出的结果列表

#### 输入项

必填：

- 模特图
- 至少一个提取部位

#### prompt 规则

这个模块不是自由 prompt，而是按部位套固定模板：

- 上衣：
  - `将图片中的上衣部分提取出来，只提取最外层穿着的上衣，不要包括衣架、里面的衣服、配饰等其他物品。生成纯白背景的产品图，保持服装的细节和质感，专业拍摄，高清效果。`
- 下装：
  - `将图片中的下装部分提取出来，只提取最外层穿着的下装（裤子或裙子），不要包括衣架、里面的衣服、配饰等其他物品。生成纯白背景的产品图，保持服装的细节和质感，专业拍摄，高清效果。`
- 鞋子：
  - `将图片中的鞋子部分提取出来，只提取脚上穿着的鞋子，不要包括其他物品。生成纯白背景的产品图，保持鞋子的细节和质感，专业拍摄，高清效果。`

#### 任务展开规则

如果用户多选部位，则按部位拆成多任务：

- 选 1 个部位 = 1 个任务
- 选 3 个部位 = 3 个并行任务

每个任务：

- `imageUrl = 模特图`
- `instruction = 部位专属 prompt`

#### 历史数据特征

- `originalPhotoId` 常见为 `white-background-conversion-上衣` / `white-background-conversion-下装` / `white-background-conversion-鞋子`
- 老数据里也存在 `unknown` 或空 prompt 的兼容记录

#### 平台实现建议

这个模块很适合抽成“部位型扇出任务”：

- 前端是一个多选框
- 后端统一拆任务
- 后续还可以扩展到帽子、包袋、配饰

### `c20` 爆款裂变

#### 产品目标

基于一张原图，通过“款式 / 颜色 / 图案”三个变化强度参数，批量裂变出多张相似但有差异的爆款图。

#### UI 结构

左侧：

- 上传原图
- 款式变化滑杆
- 颜色变化滑杆
- 图案变化滑杆
- 生成数量选择，支持 `1-10`
- 输出尺寸比例

右侧：

- 裂变结果列表
- 批量选择与批量下载

#### 输入项

必填：

- 原图

可选：

- 款式变化强度
- 颜色变化强度
- 图案变化强度
- 生成数量
- 尺寸比例

#### prompt 规则

该模块不依赖自由文本，而是由 3 个滑杆动态拼装 prompt：

- 款式变化：
  - `>=40`：`明显改变款式`
  - `>=25`：`适度改变款式`
  - `>=10`：`细微改变款式`
  - 其他：`几乎保持原款式`
- 颜色变化：
  - `>=80`：`大幅改变颜色`
  - `>=60`：`明显改变颜色`
  - `>=40`：`适度改变颜色`
  - `>=20`：`细微改变颜色`
  - 其他：`几乎保持原颜色`
- 图案变化：
  - `>=80`：`大幅改变图案`
  - `>=60`：`明显改变图案`
  - `>=40`：`适度改变图案`
  - `>=20`：`细微改变图案`
  - 其他：`几乎保持原图案`

最终 prompt 结构固定为：

- `根据原图进行裂变调整：`
- `款式变化 / 颜色变化 / 图案变化`
- 必须保持脸部、表情、眼神、姿势、背景和整体构图不变
- 只允许对服装进行变化

#### 任务展开规则

任务数：

- `生成数量`

每个任务：

- 使用同一张原图
- 使用同一条动态 prompt
- 只更换 `resultId` 与 `originalPhotoId`

#### 任务映射

- `imageUrl = 原图`
- `instruction = 动态裂变 prompt`
- `aspectRatio = 用户选择的比例，可为空`
- `originalWidth / originalHeight = 原图尺寸按比例裁定后的输出尺寸`

#### 历史数据特征

- `originalPhotoId` 常见为 `burst-variation-upload-0/1/2...`
- `modelId` / `styleId` 为空

#### 平台实现建议

这个模块本质是“参数驱动型批量裂变器”：

- 只需要维护参数到 prompt 的映射关系
- 任务拆分、处理中记录和轮询逻辑可直接复用批量框架
- 点数按输出张数线性扣减

### `c21` 服装换料

#### 产品目标

把一张面料图应用到一组上装 / 下装图片上，只改变服装的面料视觉特征，不改变版型和结构。

#### UI 结构

左侧：

- 上传上装，可选，支持多选
- 上传下装，可选，支持多选
- 上传面料图，必填
- 自定义提示词输入框
- 输出尺寸比例

右侧：

- 换料结果列表
- 批量选择与批量下载

#### 输入项

必填：

- 面料图
- 至少一张上装或下装图片

可选：

- 上装数组
- 下装数组
- 自定义提示词
- 尺寸比例

#### prompt 规则

当前前端实际使用的是固定 prompt：

- `请将提供的面料应用到这件衣服上，改变衣服的颜色或风格，但保持衣服的版型、款式、细节等其他部分完全不变。`
- 并追加 5 条要求：
  - 替换原有面料
  - 保持版型、剪裁、设计不变
  - 保持纽扣、拉链、口袋、装饰等细节位置和样式不变
  - 只改变颜色、图案、纹理等视觉特征
  - 输出专业电商图

#### 现有前端限制

UI 上存在“自定义提示词”输入框，但当前生成逻辑没有把该字段拼入 `instruction`。

这意味着：

- 如果严格复刻现站行为，可以保留输入框但忽略其内容
- 如果要做成更合理的平台版本，应该在新平台里明确二选一：
  - 删除这个输入框
  - 或真正接入 `buildPrompt`

#### 任务展开规则

任务数：

- `上装数量 + 下装数量`

每个任务：

- 一件服装图对应一条任务
- 全部任务共用同一张面料图和同一条固定 prompt

#### 任务映射

- `imageUrl = 单件服装图`
- `fabricImageBase64 = 面料图`
- `instruction = 固定换料 prompt`
- `aspectRatio = 用户选择的比例，可为空`
- `originalWidth / originalHeight = 每件服装图按比例裁定后的输出尺寸`

#### 历史数据特征

- `originalPhotoId` 常见为 `fabric-change-top-0` / `fabric-change-bottom-0`
- `modelId` / `styleId` 为空

#### 平台实现建议

这是很典型的“一拖多”批量编辑模块：

- 前端把上装和下装合并成统一任务数组
- 后端不关心服装类型，只处理单图 + 面料图
- 点数与输出件数一一对应

### `c23` 复刻姿势

#### 产品目标

让原图人物复刻参考图的姿势与动作，同时保持服装细节与人物身份不变。

#### UI 结构

左侧：

- 上传人物图，必填
- 提示词输入框，可选
- 上传参考人物图，可选

右侧：

- 结果列表
- 单张下载 / 批量下载 / 删除

#### 输入项

必填：

- 人物图

- 自定义模式：

- `只需要复刻姿势其他不要变`

- 打开 `useRawPrompt` 后，可以扩成更自由的人体编辑模块
- 后续还可以接入 `pose-references` 资源，沉淀姿势参考库

### `c9` 历史记录

#### 产品目标

集中查看所有分类生成结果，并提供搜索、筛选、批量下载、删除和大图预览。

#### UI 结构

顶部：

- 返回按钮
- 总记录数
- 全选 / 取消全选
- 批量下载

筛选区：

- 按分类名称搜索
- 按分类过滤
- 按时间过滤：全部 / 今天 / 最近一周 / 最近一月

列表区：

- 卡片网格
- 若存在 `originalImageUrl`，则原图与结果图并排展示
- 结果图支持点击放大预览

#### 数据来源与行为

该模块不是生成模块，而是直接读取历史结果：

- `GET /api/generated-results`
- 前端会把相对路径图片 URL 统一补成服务器完整 URL
- 同时读取：
  - `generatedImageUrl`
  - `originalImageUrl`
  - `prompt`
  - `categoryId`
  - `modelId`
  - `styleId`
  - `createdAt`

#### 功能规则

- 搜索是按分类名称匹配，不是按 prompt 搜索
- 分类过滤直接按 `categoryId`
- 时间过滤分为：
  - 今天
  - 最近一周
  - 最近一月
- 支持多选卡片批量下载
- 支持单条删除：
  - `DELETE /api/generated-results/:id`

#### 点数规则

- 历史记录模块本身不扣点
- 只消费已有 `generated-results` 数据

#### 平台实现建议

`c9` 应被视为平台基础能力，而不是附属页面：

- 所有生成模块都依赖它做结果沉淀和回看
- 应首期和 `generated-results` 数据表一起交付
- 建议后续补充更多筛选项：
  - 任务状态
  - 点数消耗
  - 原图 / 结果图下载打包
